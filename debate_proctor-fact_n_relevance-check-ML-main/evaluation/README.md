# Evaluation Harness

This directory contains the evaluation tooling for the debate fact-and-relevance checker.

## What's in here

- `dataset.json` — 30 hand-labeled fact-check examples (15 per topic) plus 5 relevance examples. Labels are based on widely-established facts in each domain.
- `run_eval.py` — runs the eval pipeline, computes metrics, saves per-example diagnostics.
- `results_<timestamp>.json` — auto-generated by each eval run (gitignored by default).

## Prerequisites

Before running eval, you need:

1. The Wikipedia corpus ingested (`python main.py --topic ai` and `python main.py --topic aadhaar`).
2. The vector index built for each topic (`python vector_engine/build_index.py --topic ai`, etc.).
3. A valid `API_KEY` for Gemini in your `.env`.

## Usage

```bash
# Full eval (both topics, fact + relevance) — ~30 examples × LLM calls
python evaluation/run_eval.py

# Fact-check only, AI topic only
python evaluation/run_eval.py --topic ai --mode fact

# Smoke test, first 3 examples only
python evaluation/run_eval.py --limit 3

# Relevance eval only (cheap, doesn't need the brain built)
python evaluation/run_eval.py --mode relevance
```

## Metrics reported

**Fact check:**
- Accuracy (overall %)
- Per-class precision, recall, F1 for SUPPORTED / CONTRADICTED / NOT_VERIFIABLE
- Confusion matrix
- Score MAE — mean absolute error of the numeric `fact_score` against the expected score for each gold label (SUPPORTED→100, CONTRADICTED→0, NOT_VERIFIABLE→50)

**Relevance:**
- Accuracy
- Per-class precision, recall, F1 for DIRECT_COUNTER / ELABORATION / TANGENTIAL / IRRELEVANT
- Confusion matrix

The system can output `MIXED` as a fact verdict (when retrieved evidence is genuinely split). For grading, `MIXED` is mapped to `NOT_VERIFIABLE` since both represent a refusal to commit to SUPPORTED or CONTRADICTED.

## Honest limitations

- **Small dataset.** 30 fact + 5 relevance examples is enough to detect regressions and report a real number, but it is not a proper benchmark. Per-class numbers on 5 examples per class have wide confidence intervals.

## A/B testing the chunker

The build pipeline now uses paragraph-aware chunking (`paragraph_aware_chunker` in `vector_engine/build_index.py`). Existing FAISS indexes that were built with the old fixed-window chunker won't reflect the change until you rebuild.

To compare:

```bash
# 1. Run eval against the current (old-chunker) index, save the number
python evaluation/run_eval.py --mode fact > eval_before.txt

# 2. Force a rebuild with the new chunker
rm data/ai/vector_store.index data/ai/metadata.pkl
rm data/aadhaar/vector_store.index data/aadhaar/metadata.pkl
python vector_engine/build_index.py --topic ai
python vector_engine/build_index.py --topic aadhaar

# 3. Re-run eval, compare
python evaluation/run_eval.py --mode fact > eval_after.txt
```

If accuracy doesn't move significantly, retrieval wasn't the bottleneck — look at the LLM verifier or the corpus coverage instead.
- **Single labeler.** All examples were labeled by one person; no inter-annotator agreement was computed.
- **Selection bias.** Examples lean toward clear-cut cases (well-known facts vs. clear falsehoods). Real debate claims are messier.
- **Corpus dependency.** Results depend heavily on what the Wikipedia ingestion pulled in. A claim labeled SUPPORTED can still be marked NOT_VERIFIABLE by the system if the relevant article wasn't crawled (or the chunk got cut mid-sentence).
- **LLM nondeterminism.** Even at low temperature, the verifier LLM can produce slightly different verdicts across runs. Numbers will vary by ±a few percentage points.

## Extending the dataset

To add an example, append to `dataset.json` with:

```json
{
  "id": "ai-016",
  "topic": "ai",
  "claim": "...",
  "expected_verdict": "SUPPORTED|CONTRADICTED|NOT_VERIFIABLE",
  "rationale": "Why this is the correct label."
}
```

Good additions: edge cases where the system currently fails, claims that test specific failure modes (e.g., subtle quantitative claims, time-sensitive facts), examples that require multi-hop reasoning across chunks.

## What "good" looks like

For a small hand-labeled set of this kind, reasonable targets are:

- Fact accuracy: ≥75% (random chance with 3 classes is 33%; a strong RAG system should be in the 75–90% range)
- Score MAE: ≤25 points
- Relevance accuracy: ≥70%

If you're well below these, look at the confusion matrix:
- Lots of SUPPORTED → NOT_VERIFIABLE means retrieval is failing (corpus gap or chunking issue).
- Lots of CONTRADICTED → SUPPORTED means the LLM verifier is too lenient (tighten the prompt).
- Lots of NOT_VERIFIABLE → SUPPORTED/CONTRADICTED means the abstain threshold is too low (raise `DECISIVE_MASS_FLOOR`).
